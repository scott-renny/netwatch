00

Terms you'll see everywhere

Every one of these gets explained the first time it appears below too. This section is here so you can come back if something feels unfamiliar.

term
Terminal
The text window where you type commands. On Ubuntu Server there's no mouse or desktop — everything is done here. You connect to it from another computer using SSH (explained below).
term
SSH
Secure Shell — a way to control your Ubuntu Server laptop from a different computer over the network. You type commands on your regular computer and they run on the server. Think of it as a remote control for the terminal.
term
IP address
A number like 192.168.1.10 that identifies a specific device on your network. Your server has one, your phone has one, every device has one. This is how devices find each other.
term
Flask / API
Flask is a Python program that turns your server into something a web browser can talk to. An API is the set of questions the browser is allowed to ask it — things like "give me the list of devices" or "turn off this profile."
term
Nginx
A lightweight program that serves web pages. When you type your server's IP into a browser, Nginx is what hands the dashboard page back to you. Without it you'd have to type a long address including a port number every time.
term
systemd / service
The system Ubuntu uses to start programs automatically. A "service" is a program registered with systemd so it starts when the server boots and restarts if it crashes. We turn NET-WATCH into a service in Section 9.
term
sudo
A word you put in front of commands that need administrator permission. Like right-clicking "Run as administrator" on Windows. The server will ask for your password the first time you use it in a session.
term
vnstat
A small program that watches how much data your server sends and receives over time, hour by hour. NET-WATCH reads from it to draw the Network Traffic chart with real numbers.
term
arp-scan
A tool that sends a network broadcast asking "is anyone there?" and records every device that answers. This is how NET-WATCH discovers devices on your network without you having to add them by hand.
term
Profile / Kill switch
A profile is a group of devices — like "Child 1" or "Gaming Devices" — that share the same schedule and internet rules. The kill switch is a button in the dashboard that instantly cuts that group's internet access.
01

What you're building

→

The dashboard

A web page that shows every device on your network in real time — which ones are online, how much bandwidth they're using, which profile they belong to, and any Wazuh security alerts. You open it in any browser on any device on your network. It fits on a tablet without scrolling.

→

The backend

A Python program running silently on your Ubuntu Server laptop. It scans the network every 30 seconds, keeps track of device assignments and profile states, and answers questions from the dashboard. You never interact with it directly after setup.

→

Profiles and time limits

You can group devices into profiles like "Kids," "Gaming," or "Work." Each profile gets its own schedule (allowed hours per day of the week) and an optional daily time limit in minutes. When the limit runs out, that profile is automatically blocked. You can override it any time from the dashboard.

ℹ️
What this does NOT do yet: The kill switch shows in the dashboard but won't actually cut anyone's internet until you connect Pi-hole in Section 12. Everything else — scanning, profiles, schedules, time limits, the traffic chart — works from the moment you finish Section 7.
02

Files you received

You were sent a folder called netwatch. Here's what each file does and why you need it.

api/netwatch_api.py
The backend brain. Scans your network, stores device and profile data, serves real-time information to the dashboard. This is the only file you'll edit to configure things.
web/index.html
The dashboard itself. This is the file you'll open in a browser — setup.sh copies it to your server and Nginx serves it at your server's IP address. One self-contained HTML file, no internet connection required after install. Fits on any screen without scrolling.
config/devices.json
Where NET-WATCH stores the list of known devices. Starts with sample data; gets replaced by your real network devices once you run the first scan.
config/profiles.json
Where profiles, schedules, and daily limits are stored. Anything you set in the dashboard gets saved here automatically.
setup.sh
A script that installs everything automatically. You run it once and it handles all the steps in Sections 4 and 9 for you.
netwatch.service
Tells Ubuntu how to run NET-WATCH as an automatic service. Used by setup.sh — you don't need to open this file.
nginx-netwatch.conf
Tells Nginx how to serve the dashboard at a plain IP address. Used by setup.sh — you don't need to open this file.

1

Option A — Use the setup script (recommended)

If you can get the netwatch folder onto your server (via USB drive, SCP, or any file transfer), the setup script does everything automatically: creates folders, copies files, installs the systemd service, configures Nginx, and opens the firewall. Run it from inside the netwatch folder.

bashubuntu server — run from inside the netwatch folder

#   scp -r /path/to/netwatch your_username@YOUR_SERVER_IP:~/
# Example:
#   scp -r ~/Downloads/netwatch scott@192.168.1.10:~/

# Then SSH into your server and run:

sudo ./setup.sh

# The script will print what it's doing at each step.
# At the end it prints your server's IP address — that's your dashboard URL.
✅
If setup.sh completes without errors, skip to Section 6. The script handles Sections 5, 7, and 9 automatically. Come back to Section 6 to configure your network settings before opening the dashboard.
ℹ️
What setup.sh actually did with your files:

It copied web/index.html → /opt/netwatch/web/index.html
It copied api/netwatch_api.py → /opt/netwatch/api/netwatch_api.py
It copied config/devices.json + profiles.json → /opt/netwatch/config/

Then it configured Nginx to watch for any browser request coming in on port 80 (the normal web port) and serve index.html in response. That's why typing your server's IP into a browser opens the dashboard — Nginx intercepts the request and hands back that file. The dashboard page then talks to netwatch_api.py (running on port 5000 in the background) to get live device data. You never have to touch index.html directly after this point.

06

Tell the backend where your network is

There are two things you must set before the dashboard will show your real devices: your network's subnet, and your server's network interface name. Everything else is optional for now.

1

Find your server's network interface name

An "interface" is what your server calls its network connection. It used to always be called eth0, but modern Ubuntu gives it names like enp2s0 or ens3. You need the real name or the scan won't work. Run this command to find it:

bashubuntu server
ip link show

# Look for an interface that ISN'T "lo" (that's just the server talking to itself)
# It will look something like:
#   2: enp2s0: <BROADCAST,MULTICAST,UP,LOWER_UP>
# The name in that example is "enp2s0" — yours will be similar

# Also find your subnet with this command:
ip addr show | grep "inet " | grep -v "127.0.0.1"
# Example output: inet 192.168.1.10/24
# Your subnet is the first three numbers + .0/24, so: 192.168.1.0/24
2

Open the backend config file

nano is a text editor that works in the terminal. The arrow keys move the cursor, Ctrl+O saves, Ctrl+X exits. You're looking for the SCAN_SUBNETS line near the top of the file.

bashubuntu server
sudo nano /opt/netwatch/api/netwatch_api.py
3

Update SCAN_SUBNETS with your real values

Use Ctrl+W to search for SCAN_SUBNETS. You'll see a line like the one below. Replace 192.168.1.0/24 with your subnet and eth0 with your interface name from Step 1.

pythonnetwatch_api.py — find this line and update it
# BEFORE (what it looks like now):
SCAN_SUBNETS = [{"subnet": "192.168.1.0/24", "interface": "eth0"}]

# AFTER (your real values — example):
SCAN_SUBNETS = [{"subnet": "192.168.1.0/24", "interface": "enp2s0"}]
#                              ↑ your subnet                     ↑ your interface name

Press Ctrl+O then Enter to save, then Ctrl+X to exit nano.

✓
I know my server's interface name (e.g. enp2s0)
✓
I updated SCAN_SUBNETS in netwatch_api.py with my real subnet and interface name
⚠️
Getting the interface name wrong is the #1 reason the device scanner finds nothing. If the dashboard shows no devices after scanning, come back here and double-check this setting with ip link show.

Make it start automatically on boot

Right now the backend only runs while you're keeping that terminal window open. This section makes it run automatically every time the server boots and restart itself if it ever crashes. Skip this section if you used setup.sh — it already did this for you.

1

Stop the manually running backend first

Go to the terminal where you started the backend in Section 7 and press Ctrl+C to stop it. We're about to hand it off to systemd.

2

Install the systemd service file

A service file tells Ubuntu exactly how to start and manage a program. Copy the one we provided and register it with systemd.

bashubuntu server
# Copy the service file into the system service directory
sudo cp ~/netwatch/netwatch.service /etc/systemd/system/netwatch.service

# Tell systemd to re-read its service files (sees the new one)
sudo systemctl daemon-reload

# Enable it — means it will start automatically at every boot
sudo systemctl enable netwatch

# Start it now (without rebooting)
sudo systemctl start netwatch

# Confirm it's running — look for "active (running)" in the output
sudo systemctl status netwatch
3

Configure Nginx to serve the dashboard

bashubuntu server
# Copy the Nginx config file
sudo cp ~/netwatch/nginx-netwatch.conf /etc/nginx/sites-available/netwatch

# Enable it (create a link in sites-enabled)
sudo ln -sf /etc/nginx/sites-available/netwatch /etc/nginx/sites-enabled/netwatch

# Remove the default Nginx page so NET-WATCH loads at the root URL
sudo rm -f /etc/nginx/sites-enabled/default

# Test the Nginx config for errors
sudo nginx -t

# If the test says "ok", reload Nginx
sudo systemctl reload nginx
4

Open the firewall for HTTP

# Check first:
sudo ufw status

# If it says "Status: active", run:
sudo ufw allow 80/tcp comment "NET-WATCH dashboard"

Set daily time limits per profile

1

Open the schedule editor

On the Users / Profiles page, click a profile card. In the edit panel, click the 📅 Edit Schedule & Daily Limits button. A full 7-day editor opens.

2

Set allowed hours and time limits

Each row is one day of the week. Three columns per row:

Start — what time internet access begins for this profile on this day.

End — what time access ends.

Limit (min) — maximum minutes of internet allowed within that window. Leave blank for no cap. Set to 60 for one hour, 120 for two hours, etc.

To block a day completely — set Start and End to the same time (e.g. both 00:00).

Click Save Schedule when done. The profile card updates immediately with the new schedule and budget bar.

3

How the budget works automatically

Every 60 seconds, the backend checks which profiles have at least one device online. For any profile that's within its allowed window, it adds 1 minute to that profile's "used today" counter. When the counter hits the daily limit, the profile is automatically blocked — no action needed from you. The counter resets to zero at midnight. You can manually reset it anytime from the dashboard if you want to grant extra time.

ℹ️
The budget tracks time by detecting whether any device in the profile is online. A device is "online" if it responded to the last automatic arp-scan (runs every 30 seconds). If a device goes to sleep, its time stops counting.

12

Connect Pi-hole — do this after Pi-hole works network-wide

ℹ️
Skip this section until Pi-hole is your network's DNS server for all devices. Without that, toggling a profile's kill switch in the dashboard changes a flag in the dashboard but doesn't actually block anyone's internet. Pi-hole must be the DNS server before the kill switch does anything real.
1

Create a group in Pi-hole for each profile

Pi-hole groups let you apply different rules to different devices. Log into your Pi-hole admin page, go to Group Management → Groups, and create one group for each profile you want to be killable. Name them exactly the same as the "Pi-hole Group Name" you'll set in Step 3 — for example: child1, gaming, guest.

Then go to Group Management → Clients, find each device's IP address, and add it to the appropriate group.

2

Test that NET-WATCH can reach Pi-hole

This is safe to run at any time — it only reads Pi-hole's version, it never changes anything.

bashubuntu server
curl http://localhost:5000/api/pihole/probe

# Good response looks like:
# {"reachable":true,"version":6,"base_url":"http://192.168.1.1:80"}
# If reachable is false, check that PIHOLE_HOST in netwatch_api.py
# matches your Pi-hole's actual IP address.
3

Enable Pi-hole in the backend config

Open netwatch_api.py with nano, find these lines near the top, and update them:

bashubuntu server
sudo nano /opt/netwatch/api/netwatch_api.py
pythonnetwatch_api.py — find and update these four lines
PIHOLE_HOST     = "192.168.1.1"      # ← your Pi-hole's IP address
PIHOLE_V6_PASS  = "your_password"    # ← your Pi-hole admin password (v6)
# OR for Pi-hole v5:
PIHOLE_V5_TOKEN = "your_api_token"   # ← from Pi-hole Settings → API
PIHOLE_ENABLED  = True               # ← change False to True
4

Set the Pi-hole Group Name on each profile

In the dashboard, go to Users / Profiles, click each profile card, and fill in the Pi-hole Group Name field with the exact group name you created in Pi-hole in Step 1 (e.g. child1). Save the profile. Now when you hit that profile's kill switch, NET-WATCH will disable exactly that group in Pi-hole — without touching any other group or Pi-hole's global ad-blocking.

5

Restart the backend to apply your changes

bashubuntu server
sudo systemctl restart netwatch

# Check that it restarted cleanly
sudo systemctl status netwatch | head -5
🚨
Important: NET-WATCH uses Pi-hole's group management API, never its global disable button. Hitting a kill switch will disable only that profile's Pi-hole group — it will NOT pause ad-blocking for your whole network. Troubleshooting

?

Dashboard won't load at all

Check the backend is running: sudo systemctl status netwatch. If it shows "failed", read the error: journalctl -u netwatch -n 30 --no-pager. Common causes: wrong Python path, missing library, file not found at /opt/netwatch/api/netwatch_api.py.

?

No devices appear after scanning

Almost always an interface name mismatch. Run ip link show and confirm the interface name in SCAN_SUBNETS matches exactly. Manual test: sudo arp-scan --interface=YOUR_INTERFACE 192.168.1.0/24 — if this returns devices, the interface name is right.

?

Traffic chart shows flat line / zeros

vnstat needs at least one hour of data. Check it's running: systemctl status vnstat. Check it has data: vnstat -h. If the interface column shows your interface name with real numbers, data exists — if the chart still shows zeros, check that the interface name in vnstat matches what you set in SCAN_SUBNETS.

?

Pi-hole probe says unreachable

Check that PIHOLE_HOST in netwatch_api.py matches Pi-hole's IP: ping 192.168.1.1. Test directly: curl http://192.168.1.1/admin/api.php. If they're on different VLANs, you need a firewall rule allowing your server's IP to reach Pi-hole on port 80.

?

Interview talking points

Why ARP scanning and not ping scanning? ARP operates at Layer 2 and works even on devices that block ICMP (ping). Devices can't ignore ARP — it's required for the network to function. This means ARP scanning finds every device on the subnet including ones that appear offline to ping.

How does Pi-hole block internet access? Pi-hole is a DNS sinkhole — it intercepts DNS queries (the "what's the IP for google.com?" requests) and returns a fake response for blocked domains. When a profile's Pi-hole group is disabled, its devices can't resolve domain names, so the internet appears broken. Direct IP connections still work — this is a DNS-layer control, not a firewall rule.

What would change if you added VLANs? The SCAN_SUBNETS list would need one entry per VLAN, each with the right interface name. Three firewall rules would be needed: server → Pi-hole on port 80, server → each VLAN subnet for ARP, and dashboard device → server on port 5000. CORS_ALLOWED_ORIGINS in the backend would be tightened to specific admin VLAN ranges.

When you add VLANs — what to update

Do this section only when you're ready to segment your network into VLANs. NET-WATCH keeps working on a flat network indefinitely — VLANs are optional. When you do add them, this section tells you exactly what breaks and how to fix it.

ℹ️
What a VLAN is, in plain English: A VLAN (Virtual LAN) splits one physical network into separate isolated networks. A device on the Kids VLAN can't talk to a device on the IoT VLAN even though they're on the same router. Each VLAN gets its own IP range — for example 192.168.10.0/24 for IoT and 192.168.20.0/24 for kids. Your router enforces the separation with firewall rules you control.
1

Before you start — note down every device's MAC address

When you move devices into VLANs, their IP addresses will change. NET-WATCH tracks devices by IP, so they'll appear as unknown after the move. MAC addresses don't change — they're burned into the hardware. Write down the MAC from the Devices table now so you can match new IPs to old device names after the transition.

Go to the Devices page in the dashboard and take a screenshot of the full table. The MAC address column is your reference.

2

Update SCAN_SUBNETS in netwatch_api.py

Add one entry per VLAN. The interface name is your server's physical network interface followed by a dot and the VLAN ID number. For example if your physical interface is enp2s0 and your IoT VLAN ID is 10, the interface name is enp2s0.10.

Check that VLAN interfaces exist on your server first:

bashubuntu server — check VLAN interfaces exist
ip link show | grep "\."
# If you see lines like "enp2s0.10" your VLAN interfaces are already created
# If nothing shows, the VLAN interfaces need to be created on your server first

# Create a VLAN interface (example — VLAN 10 on enp2s0):
sudo ip link add link enp2s0 name enp2s0.10 type vlan id 10
sudo ip link set enp2s0.10 up
# Repeat for each VLAN ID you're using
pythonnetwatch_api.py — update SCAN_SUBNETS (Ctrl+W to search in nano)
# BEFORE — single flat network:
SCAN_SUBNETS = [{"subnet": "192.168.1.0/24", "interface": "eth0"}]

# AFTER — one entry per VLAN (use your real subnet and interface names):
SCAN_SUBNETS = [
    {"subnet": "192.168.1.0/24",  "interface": "enp2s0"},      # Main LAN
    {"subnet": "192.168.10.0/24", "interface": "enp2s0.10"},   # IoT VLAN
    {"subnet": "192.168.20.0/24", "interface": "enp2s0.20"},   # Kids VLAN
    {"subnet": "192.168.30.0/24", "interface": "enp2s0.30"},   # Guest VLAN
]
3

Add firewall rules so the scanner can reach each VLAN

VLANs are isolated by design — your server can't talk to them without explicit permission. You need three firewall rules per new VLAN. Add these on your router or firewall (OPNsense, pfSense, or whatever you're using). Substitute your server's real IP and your real VLAN subnets.

firewall rulesadd these on your router — repeat for each VLAN
# Replace 192.168.1.10 with your NET-WATCH server's IP
# Replace 192.168.10.0/24 with the VLAN subnet
# Replace 192.168.1.1 with your Pi-hole's IP

# Rule 1 — server can send ARP probes into the VLAN (device discovery)
Allow: 192.168.1.10 → 192.168.10.0/24   protocol: any

# Rule 2 — VLAN devices can open the dashboard in a browser
Allow: 192.168.10.0/24 → 192.168.1.10   port: 80

# Rule 3 — server can call Pi-hole's API (kill switches + blocklists)
Allow: 192.168.1.10 → 192.168.1.1       port: 80
4

Restart NET-WATCH to pick up the new subnets

bashubuntu server
sudo systemctl restart netwatch

# Watch the first scan happen live — devices on new VLANs should appear
journalctl -u netwatch -f
5

Re-identify devices that moved to new VLANs

After the first scan, devices that moved to new VLANs will appear twice in the Devices page — once as their old IP (showing Offline) and once as a new "Unknown (new IP)" entry. Use the MAC address screenshot you took in Step 1 to match them up. Then:

— Rename the new "Unknown" entry to the device's real name

— Assign it to the correct profile

— Delete the old offline entry with the old IP

The dashboard updates immediately after each change.

6

Update Pi-hole group names if they changed

If you reorganised Pi-hole groups as part of the VLAN setup, go to Users / Profiles, click each profile card, and update the Pi-hole Group Name field to match the new group name. The kill switch and per-profile blocked sites both use this field to find the right Pi-hole group.

7

Tighten CORS once VLANs are working

Right now the backend accepts requests from any IP address (CORS_ORIGINS = "*"). Once you have VLANs, you can restrict this to only your admin VLAN so devices on the IoT or guest VLAN can't reach the dashboard API even if they can load the page.

pythonnetwatch_api.py — optional: tighten CORS to your admin VLAN only
# BEFORE (open to everything):
CORS_ORIGINS = "*"

# AFTER (only your admin LAN and VPN can reach the API):
CORS_ORIGINS = [
    "http://192.168.1.0/24",    # Main admin LAN
    "http://10.0.0.0/24",        # WireGuard VPN subnet
]
8

Update the Pi-hole password if it changed

If you changed your Pi-hole admin password during the VLAN setup, update it in the systemd service file (not in netwatch_api.py directly — environment variables are safer).

bashubuntu server
sudo nano /etc/systemd/system/netwatch.service
# Find the line: Environment=PIHOLE_V6_PASSWORD=...
# Update the password, save, then:
sudo systemctl daemon-reload
sudo systemctl restart netwatch

# Confirm Pi-hole is still reachable with the new password:
curl http://localhost:5000/api/pihole/probe
💡
Test order that saves the most time: Do Step 2 (config update) and Step 3 (firewall rules) before physically moving any devices into VLANs. That way the scanner is already watching the new subnets the moment the first device lands on them — you'll see it appear in the dashboard in real time rather than having to debug why it's not showing up.
⚠️
Wi-Fi name and password changes have no effect on NET-WATCH — it operates at the IP/MAC layer, not the Wi-Fi credential layer. Change your SSID and password freely. The only side effect is devices temporarily losing their connection and getting new IPs from DHCP, which shows them as unknown until the next scan identifies them by MAC.
